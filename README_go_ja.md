# Go 言語バージョン別主要機能・進化の歩み

本ドキュメントは、Go 言語のリリース 1.0 から最新の 1.26 までの歴史における、
主要な機能追加および改善点をまとめたものです。

## 概要

Go は「後方互換性の維持」を最優先しつつ、
6 ヶ月ごとのリリースサイクルで着実に進化を続けています。
言語仕様の拡張だけでなく、
標準ライブラリの拡充やランタイム（GC・スケジューラ）の最適化に
多くのリソースを割いている点が特徴です。

## Go らしさ（Go-way）の変遷：SWE が意識すべきこと

Go はシンプルさを追求してきましたが、近年はその「シンプルさ」の定義が拡張されています。

| 時代 | 設計思想の核 | SWE の設計への影響 |
| :--- | :--- | :--- |
| **Generics 前 (〜1.17)** | ボイラープレートを厭わない | `interface{}` とコピペが主流 |
| **Generics 後 (1.18〜)** | 型安全な再利用性 | 汎用データ構造を型安全に記述可能に |
| **Modern Go (1.21〜)** | 標準ライブラリの再定義 | `slog`/`slices` で標準化が進行 |

## 歴史的転換点

### 1. Go 1.5 — GC 革命・self-hosting

GC が並行化され、実質的に現在の GC の基礎が完成しました。
また Go コンパイラが cgo なしで self-hosting できるようになり、
「Go v2 レベルの転換点」と言えるリリースです。

### 2. Go 1.7 — context 標準化

`context` パッケージが標準ライブラリに昇格しました。
以降、Go の並行処理設計において context は不可欠な要素となり、
キャンセル伝搬・タイムアウト・値の引き回しの標準手段として定着しています。

### 3. Go 1.11 / 1.16 — Modules による依存管理革命

Go 1.11 で Go Modules が初登場し、長年の依存管理問題に決着がつきました。
Go 1.16 で Modules がデフォルト化され、
`GOPATH` ベースのワークフローは完全に過去のものとなりました。

### 4. Go 1.18 — Generics（史上最大の言語拡張）

Go の歴史において最大の言語仕様変更。
型パラメータが導入され、コードの再利用性が劇的に向上しました。
同時に fuzzing も標準機能として組み込まれています。

### 5. Go 1.26 — Green Tea GC

Go 1.25 で実験導入された Green Tea GC がデフォルト有効化され、
GC オーバーヘッドが 10–40% 削減されました。
Go の GC は 1.5 の並行化以来、段階的に改善されてきましたが、
Green Tea GC はその中でも最も大きな飛躍です。

## 実務的に重要な分岐点（覚える価値あり）

| リリース | イベント | なぜ重要か |
| :--- | :--- | :--- |
| **1.5** | GC 革命 | 並行 GC で実用的な低レイテンシを実現 |
| **1.7** | context 標準化 | 並行処理のキャンセル・タイムアウト設計の要 |
| **1.11** | Modules 導入 | 依存管理の革命、`GOPATH` からの脱却 |
| **1.16** | Modules デフォルト化 | 全プロジェクトで Modules が標準に |
| **1.18** | Generics | 型パラメータ導入、史上最大の言語拡張 |

この 5 つだけ押さえれば、Go の進化の骨格はほぼ理解できます。

## 活用方法

- **公式リリースノート:** 最新の変更詳細については
  [Go Release Notes](https://go.dev/doc/devel/release) を参照してください。
- **本リポジトリのデモ:** `go1.26/` ディレクトリに
  Go 1.26 新機能のコンパイル・実行可能なサンプルコードがあります。
- **マイグレーション:** 特定のバージョンへ上げる際は、
  `go fix` ツールや公式ブログの解説を確認することを強く推奨します。
